# Corpus Pretraining for Domain-Optimized Inference

## Concept Document

**Status:** Draft for integration with broader platform spec
**Owner:** TBD
**Last updated:** 2026-05-26

## 1. Summary

This concept proposes shifting compute from online inference to offline corpus preparation, such that smaller, cheaper LLMs at inference time can match or exceed the quality of much larger generalist models operating over raw retrieval. The mechanism is a two-level optimization loop in which an expensive LLM, orchestrated via DSPy, prepares a highly structured corpus and a jointly optimized toolset that smaller inference-time models can exploit.

The core insight: when we care primarily about *our own data and knowledge*, we should not pay general-purpose LLM compute prices for general-purpose capability. Instead, we should front-load the expensive reasoning into corpus preparation, where its results compound across every future query.

## 2. Motivation

Standard RAG pipelines amortize little. Each user query incurs the full cost of retrieval, ranking, and generation against a corpus that has been minimally preprocessed (chunking, embedding). The LLM at inference time is asked to be simultaneously a *generalist* (broad world knowledge) and a *specialist* (reasoning over our domain) — and we pay for both, every time.

The opportunity:

- Most of our value lies in reasoning over a bounded, domain-specific corpus.
- Domain structure is stable; user query patterns are largely predictable from the domain model.
- Expensive LLM compute spent once, offline, can be reused across every subsequent inference.
- Graph RAG already outperforms vector-only retrieval by encoding relationships explicitly; we can push this further by anticipating *which* relationships matter.

## 3. Architecture

### 3.1 Two-Loop Optimization

**Outer loop (offline, infrequent, expensive):**
A large frontier LLM, orchestrated through DSPy, optimizes:

- the structure and contents of the prepared corpus,
- the configuration of retrieval and graph-traversal tools,
- the signatures and prompts of tools that inference-time models will call.

The optimization target is the performance of the *inner loop* against a domain-specific evaluation suite.

**Inner loop (online, per-query, cheap):**
Smaller LLMs (selected via DSPy's optimizer against the eval suite) use the same toolset over the prepared corpus to serve user queries. The inner loop is fast and inexpensive because the expensive reasoning has already been baked into the corpus and tool definitions.

Crucially, both loops share the same tool interfaces. Improvements made by the outer loop — better graph structure, tighter tool signatures, smarter chunk annotations — flow directly into inner-loop capability without code changes.

### 3.2 The Synthetic Dataset as Forcing Function

The pipeline is driven end-to-end by a **synthetic evaluation dataset** representing the questions, tasks, and reasoning patterns characteristic of the target domain. This dataset:

- Is generated by the expensive LLM from the domain model and a sampling of the corpus.
- Serves as the eval/regression suite for DSPy optimization at both loops.
- Constitutes the optimization target — everything in the offline pipeline is shaped to maximize inner-loop performance against it.
- Can be refreshed when the domain evolves, the corpus grows, or production telemetry reveals gaps.

Without this artifact, the offline pipeline has no objective; with it, every component (chunking strategy, graph schema, tool signatures, retrieval config) becomes a tunable variable in a well-defined optimization problem.

## 4. Framework / Checklist

The concept generalizes to any domain. For a new corpus, the pipeline is:

1. **Characterize the domain.** What is the knowledge structure? What entities, relationships, and processes matter?
2. **Anticipate usage.** What kinds of questions, tasks, and reasoning will users in this domain bring?
3. **Generate the synthetic dataset.** Use a large LLM to produce a representative, weighted set of queries and expected reasoning traces.
4. **Run the outer optimization loop.** Use DSPy + large LLM to jointly optimize:
   - corpus representation (chunking, entity extraction, relationship weighting),
   - graph schema and traversal heuristics,
   - retrieval configuration (e.g., Apache Solr parameters, hybrid weighting),
   - graph database setup (e.g., FalkorDB schema and indices),
   - tool signatures and prompts.
5. **Select and configure the inference-time model.** Use DSPy to pick the smallest model that meets the eval threshold over the prepared corpus.
6. **Deploy.** Inner loop serves queries.
7. **Refresh on demand.** Re-run the outer loop when (a) corpus material is added in volume, (b) telemetry shows degradation on specific query classes, or (c) on a scheduled cadence.

## 5. Key Components

| Component | Role | Notes |
|---|---|---|
| Domain model | Defines entities, relationships, query patterns | Input artifact; can be partially inferred from the corpus by the outer-loop LLM |
| Synthetic eval dataset | Forcing function for all optimization | Refreshable; versioned |
| Outer-loop orchestrator | Expensive LLM + DSPy + tools | Runs offline, ideally idempotent given a fixed corpus + domain model |
| Prepared corpus | Chunked, entity-annotated, graph-linked | Output of outer loop; consumed by inner loop |
| Graph RAG layer | Traversal-based retrieval | Outperforms vector-only retrieval by exploiting precomputed relationships |
| Shared toolset | Same tools used in both loops | Tool *signatures* are themselves optimization targets |
| Inner-loop runtime | Small LLM + DSPy-optimized prompts + shared tools | Cheap, fast, domain-capable |

## 6. Why This Works

- **Compute amortization.** Expensive offline work is paid once and reused across every inference.
- **Specialization without fine-tuning.** Smaller models become domain-capable not because they are smarter, but because the context they receive is curated, structured, and pre-reasoned.
- **Tool-model co-optimization.** Because DSPy treats tool signatures as part of the optimization surface, the tools themselves evolve to suit the inference-time model.
- **Model portability.** DSPy's eval-driven optimization means the inference-time model can be swapped (or downgraded for cost) without rebuilding the pipeline — only the inner-loop optimization is re-run.
- **Domain focus.** No compute is spent making the system competent on knowledge outside the domain of interest.

## 7. Integration Points with Broader Platform

The following are anticipated handoffs to other parts of the platform spec; details to be reconciled:

- **Corpus ingestion pipeline.** Where does raw corpus material enter, and how is the outer loop triggered?
- **Domain model representation.** Format and versioning of the domain model artifact.
- **Eval/regression infrastructure.** Storage, versioning, and execution of synthetic datasets; gating of deploys on eval results.
- **Tool registry.** Tool signatures are shared artifacts between loops; the platform needs a single source of truth.
- **Model selection / routing.** Mechanism by which the platform substitutes inference-time models per domain.
- **Telemetry and refresh triggers.** How production observations feed back into outer-loop re-runs.

## 8. Open Questions

- **Refresh cadence and triggers.** On-demand, scheduled, telemetry-driven, or all three? What is the cost/benefit curve?
- **Cross-domain reuse.** Can graph schemas, tool signatures, or prompt patterns transfer across related domains, or is each pipeline necessarily bespoke?
- **Synthetic dataset quality assurance.** How do we verify that the synthetic dataset truly represents production query distribution? Periodic comparison against real traffic?
- **Versioning and rollback.** Prepared corpora are large artifacts; what is the storage and rollback story when an outer-loop run regresses quality?
- **Outer-loop convergence.** How do we know when the outer loop has finished — eval plateau, budget cap, manual review?
- **Boundary with general knowledge.** Some queries will require knowledge outside the domain. How does the inner loop recognize and handle these — escalate to a larger model, decline, or hybrid retrieval?

## 9. Next Steps

3. Measure: inner-loop quality vs. cost, against a baseline of a large generalist model over standard RAG.
4. Reconcile integration points (Section 7) with the broader platform spec.
