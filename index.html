Opportunistic Verifiable Inference Runtime Network

A runtime for inference that repeats.

OVIR.net targets the workload shape where many short questions repeatedly hit the same long corpus. It fragments the corpus, routes each query to likely evidence, reuses cached fragment work, escalates uncertain cases, and emits receipts for what ran.

The repeated-inference pattern

The sparse path is accepted only when local evidence is strong. Low confidence, global comparison, adversarial ambiguity, or high-value decisions trigger a fallback whose answer becomes reusable network memory.

Sparse attention

BigBird analogy

Use local, random, and global structure to avoid treating every token pair as equally relevant.

Sparse experts

PEER analogy

Use product-key retrieval to activate a small number of experts or fragment processors instead of a dense path.

Runtime memory

Fallbacks cache too

An expensive fallback is not discarded; it becomes a receipt-backed answer or routing boundary for similar future queries.

Runtime loop

The runtime is a graph of small verifiable jobs rather than a single model call. The unit of scheduling is the fragment; the unit of trust is the receipt.

Ingest stable corpushash corpus version, tokenizer, model snapshot, precision, and fragment boundaries.
Route short queryscore likely fragments with sparse retrieval, product-key expert lookup, or cached summaries.
Execute selected fragmentsrun ONNX/Nx fragment work only where evidence is likely and cache is cold.
Merge and decideaccept sparse result only when confidence, citation coverage, and conflict checks pass.
Fallback when neededrun a heavier model, reranker, full-context path, or human review for global or uncertain queries.
Emit receiptrecord hashes, worker IDs, cache hits, fallback calls, latency, cost, and spot-recompute checks.

Technical spine

The implementation plan keeps the stack narrow: sparse attention, sparse expert routing, portable graphs, distributed workers, shared cache, and deterministic receipts.

Attention

BigBird masks

Nx tensors implement full and sparse attention so the approximation gap is measurable.

Experts

PEER routing

Product-key lookup activates top-k expert fragments without brute-force scoring over every expert.

Execution

ONNX + Ortex

Expert fragments are portable graph files; model bytes define the stable identity.

Distribution

Elixir / OTP

GenServer fragment workers, Supervisors, Registries, ETS cache, and BEAM nodes map directly to OVIR.

Build target

The first falsifiable target is a small integration benchmark, not a production network.

Week 5 integration test

Document: 200 tokens, 20 fragments of 10. Experts: 64 with product-key top-2 routing. Workers: 20 FragmentWorker GenServers under a Supervisor. Queries: 100 total from 10 unique queries repeated 10 times. Passing conditions: cache hit rate above 80%, fallback rate below 30%, expert calls below dense execution, output shape stable, and every receipt valid.
