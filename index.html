Opportunistic Verifiable Inference Runtime

Stop recomputing neural work.

OVIR is a content-addressed execution control plane for high-volume inference workloads where recomputation, cache locality, KV movement, embedding lookups, and tail latency dominate cost.

Workloads first

OVIR starts where inference spend concentrates: repeated prefixes, large KV caches, recommendation ranking, embeddings, adapter-heavy serving, and batch workflows that can reuse intermediate neural computation.

01 / LLM serving

Long-context KV and prefix reuse

Repeated system prompts, RAG context, agent history, and document prefixes should become reusable execution artifacts.

Pain
Prefill cost, KV transfer, hot-prefix recomputation.
Lever
Content-addressed KV blocks and prefix-aware routing.
02 / Recsys

Ads, feeds, ranking, and retrieval

Ranking systems depend on huge embedding tables, sparse feature access, model sharding, and strict p99 latency.

Pain
Random memory access, shard imbalance, tail utilization.
Lever
Feature locality, hot embedding reuse, adaptive placement.
03 / RAG

Embeddings and document pipelines

Enterprise document workloads repeatedly embed, rerank, summarize, and prefix-cache the same corpus fragments.

Pain
Duplicate preprocessing and repeated chunk inference.
Lever
Hash-addressed embeddings, reranker outputs, summaries.
04 / Model fleets

Adapters, LoRAs, and model variants

Multi-tenant inference stacks waste memory and time on cold starts, fragmented variants, and poor batching.

Pain
VRAM fragmentation, adapter churn, under-filled batches.
Lever
Adapter-locality routing and compatible-batch scheduling.

Toy model: Bertlet Butterfly

A falsifiable first demo: distill a frozen BERT-style teacher into many tiny shared-weight encoders over token blocks, then propagate block summaries through an FFT-style butterfly. The aim is not exact attention equivalence; it is a cached, distributed approximation that keeps BERT-like task quality while making cross-token communication logarithmic.

Baseline

One dense BERT teacher

Run a standard encoder over N tokens and use full self-attention in every layer. This is the quality target and the source of distillation labels, logits, and hidden-state targets.

attention/layer ≈ O(N² · d)
memory/layer ≈ O(N²)
artifact = whole sequence state
Decompose

Many small Bertlets

Split the sequence into B = N / b fixed-size blocks. Run the same small encoder on each b-token block. Cache every block output by model, tokenizer, block text, layer, precision, and config hash.

local work ≈ B · O(b² · d)
if b is fixed: O(N · b · d)
artifact = hash(block output)
Mix

FFT-style summary exchange

Each block emits s summary tokens. A butterfly network exchanges summaries across log₂B rounds, giving every block a path to every other block without materializing an N×N attention matrix.

global rounds = log₂B
summary work ≈ O(B · logB · s² · d)
target = O(N logN) communication
Concrete settingN = 512 to 8,192 tokens; b = 32 or 64; B = 16 to 128 blocks; s = 2 to 8 summaries per block.
Training signalHard labels plus KL distillation from teacher logits, optional hidden-state matching at selected layers, and contrastive loss for reranking tasks.
Pass conditionRecover at least 97% of BERT-teacher task score on short contexts, exceed dense attention throughput on long contexts, and show cache-hit savings on repeated corpora.
Fallback pathRoute high-uncertainty examples, global-dependency-heavy tasks, or final rerank candidates back to the full teacher or a hybrid top-attention layer.
Hash the inputsmodel snapshot, tokenizer, block text, block index, adapter, precision, runtime config.
Run Bertlets in paralleleach block is an independent work unit that can execute on cheap, local, or already-warm workers.
Butterfly merge summariesround 1 mixes neighbors, round 2 mixes stride-2 blocks, round k mixes stride-2ᵏ blocks.
Cache every intermediateblock states, summary states, merge outputs, logits, embeddings, and receipts become reusable artifacts.
Verify selectivelyspot-check cached artifacts by recomputation; compare output hashes, logits drift, and latency/cost deltas.

Concrete use cases

The runtime is intentionally narrow: it does not generate code or replace inference engines. It decides what can be reused, where work should run, and how results are verified.

KV cache

Prefix-aware LLM router

Route requests to workers already holding matching model, tokenizer, prompt-prefix, and KV-block state.

Prefill / decode

Disaggregated serving policy

Choose whether prefill, decode, KV transfer, or cache reuse is cheapest for the current request shape.

Ranking

Embedding-shard scheduler

Group ranking requests by sparse feature overlap, shard locality, model snapshot, and latency target.

Batch inference

Reusable document work

Cache embeddings, reranker scores, summaries, and model-prefix states for repeated document pipelines.

Model variants

Adapter placement

Keep compatible LoRAs and fine-tunes close to shared base weights, warm replicas, and batchable traffic.

Audit

Recompute avoidance receipts

Emit hashes, worker metadata, cache-hit evidence, latency, cost, and execution lineage for each result.

Depth of the contemplated stack

OVIR sits above execution engines and below applications: a scheduler, content-addressed store, cache fabric, and verification layer that can integrate Ray, MAX, Mojo, ONNX, Elixir, Nx, and REE-style receipts.

Distributed substrate

Ray tasks and actors; BEAM/Elixir supervision; Kubernetes or VM swarms for placement and failure recovery.

Execution engines

MAX, Mojo kernels, ONNX Runtime, vLLM, SGLang, Nx, Ortex, or other node-local inference backends.

Content identity

Unison-inspired hash addressing for models, graph fragments, prompts, tensors, KV blocks, kernels, and outputs.

Cache fabric

CAS over local disks, object stores, GPU memory, CPU RAM, and network-transfer-aware KV movement.

Receipts

Gensyn REE-inspired reproducibility boundary plus telemetry for what ran, what was reused, and what was avoided.

Runtime loop

Every reusable fragment becomes an execution unit: deterministic where possible, hashable by construction, schedulable by locality, and auditable after completion.

Normalize the requestModel, weights, tokenizer, prompt, input tensors, adapter, precision, runtime config.
Derive execution unitsKV blocks, embedding lookups, graph partitions, layer ranges, tensor shards, microbatches, Bertlet blocks.
Resolve from content storeReturn existing artifacts when hashes, version constraints, and equivalence checks match.
Place by localitySchedule work near cached weights, hot KV blocks, embedding shards, adapter state, or cheaper hardware.
Execute with backend adaptersUse MAX, ONNX Runtime, vLLM, SGLang, Nx, Mojo, or REE-wrapped workers where appropriate.
Record lineagePersist output hashes, cache hits, transfers, worker metadata, latency, and recomputation avoided.
