Opportunistic Verifiable Inference Runtime Network

A network for repeat long-context inference.

OVIR.net fragments one workload pattern: many short queries repeatedly executed against the same long context. The network places those fragments near the right context, cache, model, or fallback answer, then returns receipts for what ran and what was reused.

The fit is narrow

OVIR.net should only be used where a stable long context is queried many times. It is not a general transformer replacement and not a shortcut for one-off dense reasoning.

Good fit

Repeated questions over one corpus

Legal discovery, contract review, due-diligence data rooms, support archives, code search, compliance manuals, clinical literature, and enterprise knowledge bases.

many Q_short × same C_long
stable chunks
reuse compounds
Bad fit

Unique prompt, unique context

Creative generation, one-off analysis, fast-changing context, or tasks where most tokens plausibly depend on most other tokens.

unique Q × unique C
little amortization

Design rule

Shortcut only when confidence is high

The network is a triage layer. It accepts local/sparse evidence when strong and routes uncertain cases to a heavier model.

cheap path = confident
fallback = uncertain
cache both

The network is the runtime

The runtime is a graph of small verifiable jobs, not a single model invocation. Each fragment can be placed where execution is cheapest or most reusable.

1 / Address

Hash the context

Chunk the long context once. Version every fragment by corpus, tokenizer, model snapshot, adapter, and precision.

2 / Place

Route the query

Send short-query work only to relevant fragments, cached summaries, warm workers, or known fallback answers.

3 / Reduce

Merge sparse evidence

Combine local scores, citations, summaries, and conflict checks into an answer or escalation decision.

4 / Verify

Issue receipts

Record hashes, worker IDs, cache hits, fallback calls, latency, cost, and spot-recompute checks.

Concrete example

A 20-token question targets a 2,000-page data room. OVIR.net does not push the full room through one model. It retrieves cached fragment summaries, scores likely evidence fragments across the network, merges the strongest evidence, and escalates only if the result is weak, conflicting, or globally dependent.

Fallbacks are network memory

A fallback is not a failure. It is an expensive answer that becomes reusable state for future routing, acceptance, and verification.

Trigger fallbacklow confidence, conflicting chunks, global comparison, synthesis, ordering, or high-value answer.
Run heavier pathuse a full-context model, larger reranker, domain model, or human-reviewed path when needed.
Cache the resultstore answer, citations, evidence set, corpus version, model snapshot, confidence boundary, and receipt.
Improve routingsimilar future queries can reuse the fallback or avoid the shortcut boundary that failed.

Minimal stack

Only five capabilities are essential: distributed placement, portable execution, content identity, shared cache, and verifiable receipts.

Placement

Elixir OTP scheduler

GenServer workers under a Supervisor tree route fragments by context locality, cache availability, cost, and trust score. Crashed workers restart automatically.

Execution

Nx + ONNX fragments

Tensor operations run via Elixir Nx (EXLA backend). Expert and attention fragments are serialized as ONNX graphs and loaded via Ortex, giving a backend-neutral execution boundary.

Memory

Content-addressed cache

ETS stores fragment summaries, scores, embeddings, fallback answers, and receipts keyed by SHA-256 of the ONNX model bytes + input hash.

Trust

Receipt verifier

Samples recomputation, compares hashes, tracks drift, and penalizes unreliable workers or artifacts.
